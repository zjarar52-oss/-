📘 Voice Calendar Assistant（语音日历工具）

“人们只有用心去看，才能看到真实。”
这个项目最初的想法很简单：能不能让日历不再依赖点击和输入，而是通过“说一句话”就完成记录。

在真实生活里，很多日程并不是“打开应用 → 输入 → 保存”这种标准流程产生的，而是在走路、上课、通勤甚至发呆的瞬间突然出现的。如果记录这些事情需要复杂操作，那么工具本身就会变成负担。

这个项目试图解决的就是这个问题：让日程管理回到最自然的表达方式，也就是语音。

项目在做什么

Voice Calendar Assistant 是一个基于 Web 的语音日历工具，它把语音作为主要输入方式，让用户可以直接通过自然语言完成日程管理。

比如你说“明天下午三点面试”，系统会自动识别时间和事件内容，然后把它变成一条真正的日程记录，并显示在日历中。

除了日程管理之外，这个项目还尝试引入“时间的第二层结构”，也就是过去的记录。生日、纪念日、高光事件这些通常被分散在不同应用中的信息，在这里会被统一放进同一个时间轴中，让日历不只是“未来提醒工具”，也变成“个人时间记录器”。

核心设计思路

整个系统围绕两个时间维度展开，一个是“当下与未来”，另一个是“过去的记忆”。

当下与未来部分承担的是最基础的日程管理功能，比如添加、查看和删除日程，同时支持语音输入，让用户在不操作界面的情况下完成记录。这个部分的目标是尽可能减少操作路径，让“说话”变成默认输入方式。

而过去的部分则更偏向个人记忆的整理，比如生日、纪念日或者某些重要经历。这些内容不会像任务一样被提醒频繁打扰用户，而是作为一种“时间标记”存在，让用户可以回看自己的生活轨迹。

如何运行这个项目

如果你已经部署到 GitHub Pages，可以直接通过链接访问。如果是在本地运行，这个项目不需要复杂环境，只需要打开 index.html 就可以正常使用。

由于这是一个纯前端项目，它不依赖任何后端服务，所以不需要安装数据库，也不需要启动服务器。

技术实现方式

语音部分使用的是浏览器自带的 Web Speech API，当用户点击语音按钮后，浏览器会直接进行语音识别并返回文本内容。

在语义理解上，目前提供了两种方式。第一种是本地规则解析，通过关键词识别时间和动作，比如“明天”“下午”“添加”等词汇来判断用户意图。这种方式适合部署在 GitHub Pages 这种纯静态环境中。

第二种方式是接入 DeepSeek 或类似的大模型 API，由模型直接完成意图识别和结构化输出。这种方式可以更好地处理复杂表达，比如“下周一之前提醒我交报告”。

日历部分则完全基于原生 JavaScript 实现，通过 DOM 动态渲染日期，并将事件绑定到对应的日期格子中。

设计上的一些取舍

这个项目刻意没有使用 React 或 Vite 这一类框架，主要原因是希望它能够在 GitHub Pages 上直接运行，不需要构建步骤，也不需要额外部署流程。

另一个重要原因是，这个项目的重点不是工程复杂度，而是交互方式的变化。语音是核心入口，而不是界面结构本身。

当前的局限性

目前版本仍然比较基础。语音识别依赖浏览器支持，不同设备上的表现会有差异。语义理解如果不接入大模型 API，会比较依赖简单规则，无法处理非常复杂的表达方式。

此外，数据目前是本地存储，没有做多端同步，也没有云端账户体系。

后续可能的方向

如果继续迭代，这个项目可以往几个方向发展。

一个是接入更强的语义模型，让语音理解更加自然，不再依赖固定规则。另一个是加入跨设备同步能力，让日历真正成为“个人时间中心”。还可以扩展提醒系统，比如通知推送或者智能提醒策略。

更进一步，还可以把“过去的记录”做成一种可视化的时间轴，让用户看到自己的生活结构，而不仅仅是任务列表。

写在最后

这个项目本质上不是在做一个传统意义上的日历工具，而是在尝试回答一个问题：
如果输入方式变得足够自然，工具本身还能复杂到哪里去？
